System Working Description

Hardware and Edge Layer: The system is built on an ESP32 DevKit V1 (Xtensa LX6 dual-core, 240 MHz) that interfaces with a suite of sensors and actuators. An ESP32-CAM module (AI Thinker) equipped with an OV2640 2-megapixel camera captures iris images for biometric authentication. Environmental monitoring is provided by a DHT11 temperature/humidity sensor on GPIO 4, an MQ-2 gas/smoke sensor on GPIO 32 (ADC), and an HC-SR04 ultrasonic distance sensor (TRIG=GPIO5, ECHO=GPIO18). A 16×2 I2C LCD (PCF8574T backpack, address 0x27, SDA=GPIO21, SCL=GPIO22) displays live system status with a 3-second rotating screen cycle, while a 3×4 membrane keypad (rows=[14,12,13,27], cols=[26,25,33]) provides PIN-based fallback authentication. A 5V relay module (GPIO 26) controls the physical door lock, and a status LED (GPIO 2) together with a push button (GPIO 15) complete the user interface. All components are powered from a 5V USB/adapter supply; the AMS1117 LDO on the ESP32 board provides a regulated 3.3V rail for the microcontroller and sensors.

Firmware State Machine and Processing Pipeline: The embedded firmware implements a finite state machine (FSM) with ten states: INIT → CONNECTING → READY → MONITORING → ALERT / ENROLLING / AUTHENTICATING → AUTHENTICATED / REJECTED → ERROR. On boot, the ConfigManager loads a per-device config.json from SPIFFS, allowing one firmware binary to serve any device in the fleet. The SensorManager continuously reads temperature, humidity, and smoke levels with exponential moving average smoothing (α=0.3) and passes normalised values to the TFLite Micro inference engine (MLInference), which classifies ambient environmental risk into NORMAL, WARNING, or CRITICAL using a 3-class MLP model (~700 parameters, ~4.2 KB). When a user presses the auth button or an enrollment command arrives via Firebase, the IrisCamera captures and averages up to five OV2640 frames, extracting a 64-element feature vector (8×8 zonal mean intensity). The BiometricManager matches this vector against SPIFFS-stored templates using normalised RMS distance, and the AnomalyDetector scores the event using a sliding window of 20 entries (composite of failure rate, score proximity, and frequency spike). The ActuatorController drives the relay and LED with state-appropriate patterns (solid ON for AUTHENTICATED, 3 fast flashes for REJECTED, SOS morse for ERROR). All telemetry is buffered in a 30-entry ring buffer to protect against WiFi outages.

Dual-Cloud Architecture — Firebase and AWS: The firmware maintains two parallel cloud connections. The FirebaseManager (Firebase_ESP_Client library) pushes ambient sensor readings every 5 seconds to /readings/{deviceId}, sign-in events to /signins/{deviceId}, heartbeat data to /devices/{deviceId}/heartbeat, and biometric enrollment records to /users/{userId} in the Firebase Realtime Database hosted in asia-southeast1. JPEG iris images are uploaded to Firebase Storage (/enrollments/ and /signins/) for cloud-side verification. The AWSIoTManager simultaneously publishes biometric events and anomaly alerts over MQTT/TLS (port 8883) with X.509 mutual-TLS authentication to AWS IoT Core in ap-south-1. AWS IoT Rules Engine routes messages from iot/+/biometric/signin to the biometric_pipeline Lambda and iot/+/biometric/enroll to the enrollment_indexer Lambda. These functions call Amazon Rekognition (SearchFacesByImage with 80% similarity threshold and IndexFaces for enrollment) and publish anomaly alerts to Amazon SNS for immediate email/SMS delivery to security personnel. An API Gateway exposes /auth and /enroll HTTP endpoints for browser-based biometric capture from the dashboard's WebcamAuth component.

Web Dashboard and Backend API: The React 18 + Vite web dashboard connects directly to Firebase RTDB via the Firebase JavaScript SDK, receiving live data through WebSocket onValue() listeners on all database paths. Pages include Overview (device cards, alert strip), Sensors (Chart.js time-series for temperature, humidity, smoke, and ML risk score), Auth (WebcamAuth iris capture + sign-in log), Users (enrolled user management with match threshold display), Alerts (anomaly timeline), Health (heap/uptime table), and Voice (streaming AI briefing). The separately deployed FastAPI backend (Python/uvicorn) provides a REST and Server-Sent Events (SSE) interface: /api/predict runs TFLite inference server-side, /api/stream delivers real-time events, /api/qr generates QR codes for fallback authentication, and /api/voice-update invokes the Google Gemini API to produce an AI-generated security briefing. The backend reads from Firebase via the Firebase Admin SDK and calls AWS services via boto3.

2.3 Circuit Notes and Component Wiring Details

DHT11 Temperature / Humidity Sensor

  • Pin 1 (VCC) → 3.3V
  • Pin 2 (DATA) → GPIO 4 + 4.7 kΩ pull-up to 3.3V
  • Pin 4 (GND) → GND
  • Avoid cables >20 cm; add 100 nF decoupling cap at VCC–GND
  • Firmware uses Adafruit DHT library with 3 retries and 100 ms delay between attempts

MQ-2 Gas / Smoke Sensor

  • VCC → 5V; GND → GND; AO → GPIO 32 (ADC)
  • Heater current ~150 mA; allow 20 s warm-up before reliable readings
  • Use 3.3V voltage divider or level-shifter on AO if directly connecting to 3.3V GPIO
  • Adjust trimpot DO threshold for alarm vs. ADC analog reading path
  • ADC range 0–4095 (12-bit); 0 = clean air, 4095 = max concentration

HC-SR04 Ultrasonic Distance Sensor

  • VCC → 5V; GND → GND
  • TRIG → GPIO 5 (direct, 3.3V signal is sufficient for 5V CMOS input)
  • ECHO → voltage divider (1 kΩ + 2 kΩ) → GPIO 18 (5V → 3.3V)
  • Send 10 µs HIGH pulse on TRIG; measure ECHO pulse width
  • Distance (cm) = pulse_width_µs / 58

16×2 I2C LCD (PCF8574T, 0x27)

  • VCC → 5V; GND → GND
  • SDA → GPIO 21 (+ 4.7 kΩ pull-up to 3.3V)
  • SCL → GPIO 22 (+ 4.7 kΩ pull-up to 3.3V)
  • I2C address: 0x27 (A0,A1,A2 all pulled high on PCF8574T)
  • Library: LiquidCrystal_I2C by Frank de Brabander

3×4 Membrane Keypad

  • Connector pin order: R0,R1,R2,R3,C0,C1,C2 (7 wires)
  • R0=GPIO14, R1=GPIO12, R2=GPIO13, R3=GPIO27
  • C0=GPIO26, C1=GPIO25, C2=GPIO33
  • All row pins set as OUTPUT, col pins as INPUT_PULLUP
  • Library: Keypad by Mark Stanley & Alexander Brevig

5V Relay Module (Door Lock)

  • VCC → 5V; GND → GND; IN → GPIO 26
  • Module includes opto-isolator and flyback diode; safe to drive from 3.3V GPIO
  • GPIO 26 HIGH = coil energised = NO contact closed (door open)
  • Override auto-expires after 5 minutes via firmware timer
  • Connect door lock mechanism to COM + NO terminals

Status LED

  • GPIO 2 → 330 Ω resistor → LED anode; cathode → GND
  • I = (3.3−2.0)/330 ≈ 4 mA (safe for GPIO)
  • Patterns: slow blink (1 Hz) = idle, fast blink (5 Hz) = active, SOS = error
  • Onboard LED on ESP32 DevKit is also on GPIO 2

ESP32-CAM + OV2640 (Biometric Node)

  • Power via 5V (not 3.3V); onboard LDO supplies camera + ESP32
  • FTDI programmer needed for flashing (no onboard USB-UART on AI Thinker)
  • GPIO 0 to GND during flash; remove for normal boot
  • Camera PWDN (GPIO32) held LOW during image capture; set HIGH to save power
  • 16 MB PSRAM on module used for frame buffer
Pin Conflict Note: GPIO 21, 22, 25, 26, 27, 32 are shared between the OV2640 camera bus and other peripherals (LCD I2C, keypad, relay, MQ-2 ADC). In firmware, the camera is powered down via PWDN (GPIO 32 HIGH) when sensor reading or keypad/LCD operations occur. The camera is only powered on during ENROLLING and AUTHENTICATING states, preventing bus conflicts.
Power Budget: ESP32 active WiFi ~240 mA, OV2640 ~50 mA, DHT11 ~1 mA, MQ-2 heater ~150 mA, HC-SR04 ~15 mA, LCD ~25 mA, Relay ~90 mA. Peak total ≈ 570 mA at 5V = 2.85 W. Use a 5V/2A USB adapter minimum.
